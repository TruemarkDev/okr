# postgres-database

## ADDED Requirements

### Requirement: PostgreSQL is the sole database backend
The application SHALL connect exclusively through the `postgresql` ActiveRecord adapter using the `pg` gem. The `mysql2` gem and all MySQL-specific configuration MUST be removed; the system SHALL NOT retain a dual-adapter or MySQL fallback path.

#### Scenario: Application boots against PostgreSQL
- **WHEN** `rails server` is started with `DB_*` environment variables pointing at a PostgreSQL server
- **THEN** the application boots, serves the login page, and `ActiveRecord::Base.connection.adapter_name` is `PostgreSQL`

#### Scenario: MySQL dependency fully removed
- **WHEN** the Gemfile, Gemfile.lock, and config files are inspected after the migration
- **THEN** no reference to `mysql2`, `adapter: mysql2`, or `libmysqlclient` remains

### Requirement: Schema is regenerated for PostgreSQL
`db/schema.rb` SHALL be regenerated by running the existing migrations against a fresh PostgreSQL database, with no hand edits. The regenerated schema MUST contain no MySQL-only options (e.g. `charset:`) and MUST preserve every table, column, index, and constraint present in the MySQL schema.

#### Scenario: Fresh database setup succeeds
- **WHEN** `rake db:create db:migrate db:seed` runs against an empty PostgreSQL server
- **THEN** all migrations apply cleanly, `db/schema.rb` contains no `charset:` options, and the seeded `admin@fluxday.io` user can sign in

### Requirement: Behavior parity with the MySQL baseline
The full Minitest suite SHALL be run on MySQL before the migration to record a baseline failure set. After the migration, the failure set on PostgreSQL minus that baseline MUST be empty.

#### Scenario: Test suite parity
- **WHEN** `rake test` runs against PostgreSQL after the migration
- **THEN** no test fails that passed in the recorded MySQL baseline

#### Scenario: DISTINCT with ordered default_scope pages still render
- **WHEN** the `/teams` and `/projects` index/show actions are exercised (the known DISTINCT + ordered-`default_scope` query shapes)
- **THEN** they respond successfully with no `ORDER BY ... is not in SELECT list`-class database error

### Requirement: Search remains case-insensitive
Ransack `*_cont` search predicates (e.g. task search by tracker id, name, or description) SHALL remain case-insensitive under PostgreSQL, matching MySQL behavior, and this MUST be locked by a test.

#### Scenario: Mixed-case search matches
- **WHEN** a task named "Quarterly Report" exists and a user searches for "quarterly"
- **THEN** the task is returned in the search results

### Requirement: Docker and CI provision PostgreSQL
Docker Compose and the GitHub Actions CI workflow SHALL provision a current, supported PostgreSQL image (replacing EOL `mysql:5.6`), with the database service health-checked before the app or test run starts, and `POSTGRES_*` environment variables replacing `MYSQL_*` ones.

#### Scenario: Docker Compose end-to-end boot
- **WHEN** `docker-compose up -d --build` runs with the example env files copied
- **THEN** the app container reaches the PostgreSQL service and serves the login page

#### Scenario: CI runs green on PostgreSQL
- **WHEN** the CI workflow runs on a branch after the migration
- **THEN** `db:create db:test:prepare` and the test suite complete against the PostgreSQL service container

### Requirement: Data migration path from existing MySQL databases
A documented, verified procedure SHALL exist for moving an existing MySQL database's data into PostgreSQL: the PostgreSQL schema is created from `db/schema.rb` (Rails authoritative), data is copied with pgloader in data-only mode, and sequences are reset. The procedure MUST NOT modify or drop the source MySQL database.

#### Scenario: Data migrated with integrity
- **WHEN** the documented pgloader procedure is run against a copy of a populated MySQL database
- **THEN** every table's row count matches the source, boolean flags (e.g. `is_deleted`) retain their values, and a subsequent record INSERT succeeds without a primary-key sequence collision
